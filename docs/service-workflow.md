# 애니워터 현장업무 앱 구성 방향

## 업무 흐름

애니워터 현장 엔지니어 업무는 크게 두 갈래입니다.

### 1. 사무실 오더 기반 업무

사무실에서 접수/배정한 뒤 엔지니어가 처리합니다.

- 설치
- 철거
- A/S

필요한 흐름:

```text
고객 요청/내부 접수
→ 사무실 오더 생성
→ 담당 엔지니어 배정
→ 방문예정일 지정
→ 엔지니어 현장 방문
→ 방문 전 고객/제품/과거이력 확인
→ 작업 보고
→ 관리자/사무실 확인
→ 완료 또는 재방문
```

### 2. 주기 기반 관리 업무

CMS가 유지되고 있는 고객을 기준으로 자동 생성합니다.

- 6개월 1회 정기점검
- 1년 1회 정기교체
- 카페는 기존 고객의 경우 6개월 필터 교체
- 최근 신규 설치 카페는 1년 필터 교체 정책으로 변경
- 음식점은 대부분 6개월 점검, 1년 교체
- 신규 설치 고객의 첫 점검/교체는 단순히 설치일 기준으로 계산하지 않고, 고객이 속한 지역의 월간 방문 루트에 맞춰 정렬

필요한 흐름:

```text
관리중 고객 추출
→ 최근 방문일/최근 필터교체일 확인
→ 다음 점검/교체 예정일 계산
→ 월간/주간 방문 대상 생성
→ 엔지니어 배정
→ 현장 보고
→ 다음 주기 자동 갱신
```

## 지역별 월간 루트 규칙

애니워터는 월마다 방문하는 지역이 정해져 있습니다. 따라서 신규 설치 후 첫 정기점검/정기교체는 단순히 6개월/12개월 뒤의 날짜가 아니라, 해당 고객의 담당 지역이 배정된 방문월에 맞춰 잡습니다.

예:

```text
신규 설치 고객 지역 = 강남
강남 정기 방문월 = 6월, 12월
정책상 교체 주기 = 1년
→ 첫 정기교체는 설치일 기준 1년 근처의 강남 방문월로 배정
```

앱에서는 설치처마다 아래 값을 저장해야 합니다.

- 담당 지역
- 정기 방문월
- 점검 주기
- 교체 주기
- 카페 정책: 기존 카페 6개월 / 신규 카페 1년
- 다음 점검 예정일
- 다음 정기교체 예정일

## 앱 메뉴 제안

### 현장 엔지니어

- 오늘의 방문
- 오늘 방문 오더
- 근처 설치처
- 설치처 이력
- 현장 보고
- 임시 설치처 추가

### 사무/고객응대

- 오더 접수
- 방문 일정
- 고객/설치처 관리
- 임시 등록 검수
- 보고서 확인

### 관리자/팀장

- 대시보드
- 엔지니어별 처리 현황
- 정기점검/교체 예정 현황
- 반복 A/S 고객
- 해지/회수 고객
- 점검 양식 관리

## 데이터 구조

### 설치처

- 관리번호
- 상호
- 고객명/계약자
- 주소
- 주소메모
- 연락처
- 휴대폰
- 오픈시간
- 고객 유형: 음식점 / 카페 / 공장 / 기타
- 고객 상태: 관리중 / 확인 필요 / 해지/회수 / 임시 등록
- 위도/경도

### 계약/결제

- 계약일자
- CMS 상태
- 계약해지일
- 제품회수일
- 월사용료
- 계약만료일

### 제품/관리

- 제품명
- 모델명
- 설치장소
- 제품상태
- 관리여부
- 설치완료일
- 렌탈 장비/필터

### 필터/주기

- 필터 1~7
- 최근교환일 1~7
- 교환예정일 1~7
- 잔여일 1~7
- 최근 필터 교체일
- 다음 정기점검 예정일
- 다음 정기교체 예정일

### 오더

- 업무 유형: 설치 / 철거 / A/S / 정기점검 / 정기교체
- 접수일
- 방문예정일
- 접수구분
- 진행상태
- 접수내용
- 처리내용
- 담당 엔지니어

오더 생성 권한:

- 신규설치: 사무실만 생성, 관리자가 엔지니어 배정
- 철거: 사무실만 생성
- A/S: 사무실만 생성
- 정기점검/정기교체: 관리중 고객과 주기 규칙을 기준으로 자동 생성

A/S 배정:

- 기본은 담당지역 엔지니어 방문
- 긴급도에 따라 근처 엔지니어가 방문 가능
- 사무실에서 정하거나 엔지니어들끼리 조율할 수 있음

### 현장 보고

- 방문 유형
- 연결 오더
- 처리 제품/필터
- 도착/종료 시간
- 위치
- 체크리스트 결과
- 사진 단계
- 문제 원인
- 조치 내용
- 고객 확인
- 다음 방문 필요 여부
- 관리자 검토 상태
- 다음 방문 재확인 항목

한 고객이 여러 제품/필터를 가진 경우:

- 한 방문 보고서에서 여러 제품을 동시에 처리할 수 있어야 함
- 보고서 안에서 제품별 체크 결과와 사진을 남길 수 있어야 함
- 예: 제품 A 필터 교체, 제품 B 누수 점검을 같은 방문 보고서에 기록

앱 동작:

```text
오늘 방문 오더 선택
→ 설치처 자동 선택
→ 방문 유형 자동 선택
→ 처리 제품/필터 체크
→ 보고서 제출
→ 연결된 오더 완료 처리
```

## 다음 방문 재확인 항목

엔지니어가 현장에서 “다음에 꼭 다시 봐야 하는 내용”을 보고서 작성 중 별도로 남길 수 있어야 합니다.

예:

- 주방 하부 미세 누수 흔적 재확인
- 수압이 낮아 다음 교체 때 수압 재측정
- 고객이 이전 설치 위치 변경 요청
- 배관 정리 필요
- 필터 하우징 변색 관찰 필요

동작 방식:

```text
보고서 작성 중 재확인 항목 체크
→ 설치처에 열린 재확인 항목으로 저장
→ 다음 방문 시 방문 전 이력 화면 상단에 표시
→ 엔지니어가 확인 후 체크하면 해결 처리
```

이 항목은 일반 메모와 다르게 닫히기 전까지 계속 설치처 이력 상단에 표시합니다.

## 고객 추천 기준

엔지니어 근처 추천에는 기본적으로 아래 고객만 표시합니다.

```text
고객 상태 = 관리중
또는 고객 상태 = 임시 등록
```

해지/회수 고객은 기본 추천에서 제외하되, 관리자와 사무 담당자는 검색할 수 있어야 합니다.

## 다음에 결정할 질문

1. 정기점검 기준은 “설치완료일 기준 6개월”인가요, 아니면 “최근 방문일 기준 6개월”인가요?
2. 정기교체 기준은 “최근 필터 교환일 기준 1년”인가요, “계약일/설치일 기준 1년”인가요?
3. 설치, 철거, A/S 오더는 누가 생성하나요? 사무실만 가능한가요, 엔지니어도 현장에서 생성할 수 있나요?
4. 엔지니어 배정은 자동 배정이 필요한가요, 관리자가 직접 배정하나요?
5. 고객이 여러 제품/필터를 가진 경우 한 방문 보고서에서 여러 장비를 처리해야 하나요?
6. 네이버 밴드는 완전히 대체할 계획인가요, 아니면 당분간 보고 완료 알림만 유지할까요?
